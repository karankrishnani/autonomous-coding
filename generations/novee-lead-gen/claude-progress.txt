**Problem Identified:** The desktop scraper was correctly calling `/api/scraper/success` on successful scrapes, but was NOT calling `/api/scraper/error` on failures. This meant the `last_error` field in `platform_connections` was never populated when scrapes failed.

**Additional Issue Found:** The `getConnectionById` and `updateConnectionStatus` functions in `apps/web/src/lib/platforms.ts` were using the regular Supabase client (subject to RLS), while the session context from API calls doesn't satisfy RLS `auth.uid()` checks. This caused the `/api/scraper/success` endpoint to fail with "No connection found for platform".

**Fixes Applied:**

1. **apps/desktop/src/main.ts** - Added call to `/api/scraper/error` in the catch block of `scrapeWorkspace()`:
   ```typescript
   // Update platform connection with error (updates last_error and last_checked_at)
   await fetch(`${WEB_APP_URL}/api/scraper/error`, {
     method: 'POST',
     headers: { 'Content-Type': 'application/json', Cookie: session.token },
     body: JSON.stringify({ platform: 'SLACK', error: result.error }),
   });
   ```

2. **apps/web/src/lib/platforms.ts** - Changed `getConnectionById` and `updateConnectionStatus` to use `createServiceRoleClient()` instead of `createClient()` to bypass RLS (user already validated at API layer).
